This comprehensive guide explains how to implement and utilize signals in the Phygrid platform to collect data from interaction points, track user behavior, and monitor application performance. Signals provide real-time comprehensive analytics and monitoring capabilities.

## **üìä Understanding Signals**

Signals are the foundation of data collection, alerts and analytics in the Phygrid platform. They enable real-time insights into user interactions, application performance, and system health. The data collected through signals is processed by the Phygrid platform and presented in the console's analytics dashboards.

### **Signal Types**

The Phygrid platform supports three distinct types of signals, each serving specific purposes:

#### **1. Client Signals**

Client signals are automatically generated when a client connects to your application. These signals capture essential client information that helps identify and categorize user interactions.

**When generated:** Automatically upon client connection
**Data captured:**

- Client IP address
- User agent information
- Device type and capabilities
- Browser information
- Geographic location (if available)

**Example use cases:**

- Track unique visitors and user sessions
- Analyze device and browser distribution
- Monitor geographic usage patterns
- Identify potential security threats

#### **2. Session Signals**

Session signals track the lifecycle of application sessions, providing insights into application usage patterns and user engagement.

**When generated:**

- Application startup (session start)
- Application shutdown (session end)
- Session timeout or interruption

**Data captured:**

- Session duration
- Application version
- Installation details
- Session start/end timestamps

**Example use cases:**

- Monitor application uptime and reliability
- Track user session duration and engagement
- Identify application crashes or unexpected shutdowns
- Analyze usage patterns across different time periods

#### **3. Event Signals**

Event signals are the most versatile and commonly used signal type. They capture specific user interactions and application events, providing detailed insights into user behavior and application functionality.

**When generated:** Manually triggered by your application code
**Data captured:**

- Event type and category
- User interaction details
- Product or content information
- Transaction data
- Custom parameters

**Example use cases:**

- Track user searches and navigation patterns
- Monitor e-commerce transactions and conversions
- Capture user feedback and ratings
- Track application errors and performance issues
- Monitor specific business processes

### **Analytics and Business Intelligence**

#### **Validation Requirements**

The signals service includes built-in validation for certain parameters:

- **tenantId**: Must be exactly 24 alphanumeric characters
- **spaceId**: Must be exactly 24 alphanumeric characters
- **clientId**: Must be 24-36 alphanumeric characters, hyphens, underscores, or equals signs
- **ip**: Must be a valid IPv4 or IPv6 address format
- **eventType**: Must be 5-60 characters, uppercase letters, numbers, and underscores only
- **sessionId**: Maximum 36 characters
- **str1-str5**: Maximum 200 characters each

**Example validation errors:**

```typescript
// ‚ùå Invalid tenant ID format
await signals.initialize({
  tenantId: "invalid-tenant-id", // Must be 24 alphanumeric chars
  // ... other required params
})

// ‚úÖ Valid tenant ID format
await signals.initialize({
  tenantId: "507f1f77bcf86cd799439011", // 24 alphanumeric chars
  // ... other required params
})
```

#### **Initialization Examples**

**Example 1: Automatic Initialization (Recommended)**

When implementing signals within a screen app or edge app deployed on a Phygrid device, all required initialization parameters are automatically retrieved from the device configuration. No manual parameter passing is necessary.

```typescript
import { connectPhyClient } from "@phygrid/hub-client"

const client = await connectPhyClient();
const signals = await client.initializeSignals();
signals.sendEvent(...);
```

**Example 2: Manual Initialization with Custom Parameters**

For scenarios where signals are implemented in standalone scripts or applications running outside of the Phygrid device environment, manual parameter configuration is required to establish proper signal initialization.

```typescript
import { connectPhyClient } from '@phygrid/hub-client';

const initParams = {
  "deviceId": "device-12345",
  "installationId": "install-67890",
  "spaceId": "space-abcdef",
  "tenantId": "tenant-xyz",
  "appVersion": "1.0.0",
  "appId": "com.example.app",
  "environment": "production",
  "dataResidency": "EU",
  "country": "SE",
  "installationVersion": "2.3.4",
  "accessToken": "eyJhbGciOiJIUzI1NiIsInR...",
  "clientUserAgent": "Mozilla/5.0 (Windows NT 10.0; Win64; x64)",
  "ip": "192.168.1.100"
};

const client = await connectPhyClient();
const signals = await client.initializeSignals(initParams);
signals.sendEvent(...);
```

**What to expect:**

- **Successful initialization** will return a `SignalsService` instance ready for use
- **Connection errors** may indicate network issues or invalid credentials
- **Configuration errors** may suggest missing or invalid initialization parameters
- **Timeout errors** may indicate network connectivity problems

## **üîß Available Methods**

The `SignalsService` provides a comprehensive set of methods for different types of signal transmission. These methods are organized into logical categories based on their functionality and use cases.

### **Instance Management**

Methods for managing signal service instances and retrieving configuration information.

#### **`getInstanceProps()`**

Returns the current instance properties and configuration.

**Returns:** `object` - Instance properties including device ID, tenant ID, and other configuration details

**Example:**

```typescript
const instanceProps = signals.getInstanceProps()
console.log("Current instance:", instanceProps)
```

### **Session Management**

Methods for creating and managing application sessions, which are essential for tracking user engagement and application lifecycle.

#### **`createSession(instanceParams?: Partial&lt;SignalInstance&gt;)`**

Creates a new session and initializes the instance with the required parameters.

**Parameters:**

- `instanceParams`: `Partial&lt;SignalInstance&gt;` (optional) - Instance parameters to override before creating session

**Returns:** `Promise&lt;void&gt;` - Resolves when session is successfully created

**Example:**

```typescript
await signals.createSession({
  deviceId: "device-12345",
  appVersion: "1.0.0",
})
console.log("New session created")
```

#### **`initialize(signalParams: InitSignalPayload)`**

Initializes the SignalsService with the provided signal parameters.

**Parameters:**

- `signalParams`: `InitSignalPayload` - Configuration parameters for signal initialization

**Returns:** `Promise&lt;void&gt;` - Resolves when initialization is complete

**Example:**

```typescript
await signals.initialize({
  deviceId: "device-12345",
  appVersion: "1.0.0",
})
```

#### **`initializeEdgeApp(signalParams: InitSignalPayload | {} = {})`**

Initializes the SignalsService specifically for edge applications with the given parameters.

**Parameters:**

- `signalParams`: `InitSignalPayload | {}` - Configuration parameters (defaults to empty object)

**Returns:** `Promise&lt;void&gt;` - Resolves when edge app initialization is complete

**Example:**

```typescript
await signals.initializeEdgeApp({
  deviceId: "edge-device-12345",
  environment: "production",
})
```

Sets a state key, value, and expiry duration for the current session.

**Parameters:**

- `params`: `object` - State parameters
  - `key`: `string` - State key identifier
  - `value`: `string | number` - State value to store
  - `expiryDuration`: `number` - Expiry duration in milliseconds

**Returns:** `Promise&lt;void&gt;` - Resolves when state is successfully set

**Example:**

```typescript
await signals.setState({
  key: "user_preferences",
  value: { theme: "dark", language: "en" },

})
```

## **üìà Best Practices**

Implementing signals effectively requires careful planning and adherence to best practices. Follow these guidelines to ensure optimal performance and data quality.

### **Signal Design Principles**

1. **Relevance:** Only send signals for events that provide meaningful insights
2. **Consistency:** Use consistent naming conventions and data structures
3. **Efficiency:** Avoid sending excessive or redundant signals
4. **Privacy:** Ensure compliance with data protection regulations
5. **Reliability:** Implement proper error handling and retry mechanisms

### **Performance Optimization**

1. **Batch processing:** Group related signals when possible
2. **Asynchronous transmission:** Use non-blocking signal transmission
3. **Rate limiting:** Implement appropriate rate limiting to prevent overload
4. **Data compression:** Minimize payload size for efficient transmission
5. **Connection pooling:** Reuse connections when possible

### **Error Handling**

1. **Graceful degradation:** Continue application operation even if signal transmission fails
2. **Retry logic:** Implement exponential backoff for failed transmissions
3. **Logging:** Maintain detailed logs for debugging signal issues
4. **Monitoring:** Set up alerts for signal transmission failures
5. **Fallback mechanisms:** Provide alternative data collection methods when needed

### **Data Quality**

1. **Validation:** Validate signal data before transmission
2. **Completeness:** Ensure all required fields are populated
3. **Accuracy:** Verify data accuracy and consistency
4. **Timeliness:** Send signals as close to the event occurrence as possible
5. **Context:** Include relevant context information with each signal
